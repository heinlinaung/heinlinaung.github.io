notes on running edx behind a tutor stack

2026-04-12 · ~8 min read · infra, edx, docker

tutor is the easiest way to run open edx in production without owning a devops team — until you need to bridge it into something that wasn't on the original blueprint. for fintech festival 2022, we needed oauth between a third-party identity provider and a tutor-hosted edx. here's what stuck.

the constraint

we couldn't fork. updates from upstream tutor land regularly, and any patch we held would rot. so the rule was: plugins or nothing. 

tutor plugins install ./our-oauth-plugin
tutor plugins enable our-oauth-plugin
tutor config save

what worked

  • treating oauth config as a tutor plugin, not a fork.
  • letting nginx terminate tls, not django.
  • k8s for the lms / lms-worker / mysql split — but only because we already had a cluster.

what i'd do differently

if you don't already run k8s, do not adopt it for one event. use docker compose on a single beefy box and sleep at night.

the k8s setup wasn't wrong — it just demanded constant attention for a workload that peaked once and went quiet. a vertical box would've been half the operational cost.

the takeaway

tutor's plugin system is the load-bearing piece. respect it, and edx becomes surprisingly tractable. patch around it, and you'll spend the rest of the year merging upstream changes by hand.
